# algo_a 测试方案

## 概述

algo_a 负责从微信 macOS 客户端中提取未读消息。整个流程依赖 Accessibility API 操控 UI，
因此测试必须覆盖 **真实 UI 交互** 中会遇到的所有状态。

测试分为三层：Basic（基本流程）、Edge Cases（边界场景）、Precision（精度与速度）。

## 1. Basic：微信未开启 / 冷启动状态下的完整读取

### 1.1 微信未在前台运行

| # | 场景 | 预期行为 |
|---|------|----------|
| 1.1.1 | 微信 app 未启动（进程不存在） | `find_wechat_window` 抛出明确错误，pipeline 不进行任何 UI 操作 |
| 1.1.2 | 微信已启动但窗口最小化到 Dock | `find_wechat_window` 能通过 pid 找到 app，调用 `AXRaise` 将窗口拉到前台后继续 |
| 1.1.3 | 微信已启动但窗口在另一个 Desktop/Space | 同上，需要切换到对应 Space 或把窗口拉过来 |
| 1.1.4 | 微信在后台运行（只有 menubar 图标，窗口已关闭） | 需要通过 `NSWorkspace` 激活 app 使其打开主窗口，再获取 window ref |

### 1.2 Accessibility 权限

| # | 场景 | 预期行为 |
|---|------|----------|
| 1.2.1 | 已授权 Accessibility | `ensure_permissions()` 通过，pipeline 正常继续 |
| 1.2.2 | 未授权 Accessibility | `ensure_permissions()` 弹出系统授权弹窗，assert 失败并终止，输出清晰提示 |
| 1.2.3 | 运行中被撤销权限 | 后续 AX 调用返回错误码，pipeline 应 crash 并报告 "permission revoked" |

### 1.3 正常流程端到端

| # | 场景 | 预期行为 |
|---|------|----------|
| 1.3.1 | 微信打开，侧边栏有 3 个未读聊天 | `list_unread_chats` 返回 3 个 `ChatInfo`，pipeline 依次点击、滚动、读取、写 JSON |
| 1.3.2 | 微信打开，没有未读聊天 | `list_unread_chats` 返回空列表，pipeline 跳过循环，返回空 `written_paths` |
| 1.3.3 | 未读聊天不在 `groups_to_monitor` 列表中 | `target_chats` 过滤后为空，不点击任何聊天 |
| 1.3.4 | 多次连续运行 pipeline | 第一次运行后未读标记消除，第二次应返回空列表（幂等性） |

## 2. Edge Cases：侧边栏未读标识的各种变体

### 2.1 红点 / Badge 类型识别

微信侧边栏的未读标识有多种视觉样式，AX 树中的表现也不同：

| # | 场景 | UI 表现 | AX 树特征 | 预期行为 |
|---|------|---------|-----------|----------|
| 2.1.1 | 普通个人聊天，有未读数字 | 红色圆形 badge，显示数字（如 "3"） | AXStaticText 子元素含数字文本 | `ChatInfo.unread_count = 3` |
| 2.1.2 | 普通个人聊天，99+ 条未读 | Badge 显示 "99+" | AXStaticText 含 "99+" | `unread_count` 设为 99 或特殊值表示溢出 |
| 2.1.3 | 群聊（未免打扰），有未读数字 | 同个人聊天的红色 badge | 同上 | 正常识别，`ChatInfo.name` 为群名 |
| 2.1.4 | **群聊已免打扰（muted），有未读** | **红色小圆点**（无数字），右上角 | Badge 元素存在但 **无 AXStaticText 子节点**，或 AXValue 为空 | 必须识别为有未读；`unread_count` 设为 -1 或 0 表示"有未读但数量未知" |
| 2.1.5 | **个人聊天已免打扰（muted），有未读** | 同上小红点 | 同上 | 同上 |
| 2.1.6 | 免打扰聊天无未读 | 无红点，无 badge | 无 badge 子元素 | 不应出现在返回列表中 |
| 2.1.7 | 服务号 / 订阅号（折叠在"订阅号消息"入口） | "订阅号消息" 行本身可能有 badge | 需要识别并决定是否展开（当前 pipeline 可跳过） |
| 2.1.8 | 系统通知（如 "微信团队""文件传输助手"） | 可能有 badge | 应正常识别，由 `groups_to_monitor` 过滤 |

### 2.2 群聊 vs 个人聊天的区分

| # | 场景 | 判断方式 | 注意点 |
|---|------|----------|--------|
| 2.2.1 | 群聊名称包含特殊字符（emoji、括号等） | 直接读 AXValue/AXTitle | 文件名写入时需 sanitize（`write_messages_json` 已处理 `/` 和 `\`） |
| 2.2.2 | 群聊和个人聊天同名 | 极端 case | 可能需要额外 AX 属性区分或加后缀 |
| 2.2.3 | 群聊名称为空（用户未设置群名） | AXValue 可能是成员列表拼接 | 需要处理超长名称截断 |

### 2.3 侧边栏滚动（超出可见区域的未读聊天）

| # | 场景 | 预期行为 |
|---|------|----------|
| 2.3.1 | 所有未读聊天在当前可见区域内 | 不需要滚动侧边栏，直接返回 |
| 2.3.2 | **部分未读聊天在可见区域下方** | 需要向下滚动侧边栏，每次滚动后重新扫描新出现的行，去重后合并 |
| 2.3.3 | 滚动后之前可见的行消失 | 已收集的 `ChatInfo` 不应丢失；用 `name` 去重 |
| 2.3.4 | 滚动到底部后没有更多行 | 检测滚动位置不再变化，停止滚动 |
| 2.3.5 | 聊天列表非常长（200+ 聊天） | 滚动循环需要有 **最大迭代次数** 防止无限循环 |
| 2.3.6 | 滚动过程中新消息到达，列表顺序变化 | 去重逻辑基于 `name` 而非位置，容忍列表重排 |

### 2.4 点击聊天行（click_into_chat）

| # | 场景 | 预期行为 |
|---|------|----------|
| 2.4.1 | 点击后右侧面板正常切换到对应聊天 | 继续 pipeline |
| 2.4.2 | 点击后出现"拉取历史消息"的 loading 状态 | 需要等待 loading 消失再读取消息 |
| 2.4.3 | 聊天行已被滚动出可见区域 | `AXPress` 可能失败；需要先滚动到该行可见再点击 |
| 2.4.4 | 点击免打扰群聊后红点消失 | 正常行为，不影响后续读取 |
| 2.4.5 | 快速连续点击多个聊天 | 需要在每次点击后等待右侧面板完成加载再进行读取 |

## 3. Precision：精确点击 + 快速滚动 + 多模态消息读取

### 3.1 精确点击未读聊天行

| # | 场景 | 要求 |
|---|------|------|
| 3.1.1 | 使用 `AXPress` action 而非坐标点击 | 优先用 AX action，避免坐标偏移问题 |
| 3.1.2 | 若必须用坐标点击 | 使用 `AXPosition` + `AXSize` 计算行中心点，**不能硬编码坐标** |
| 3.1.3 | Retina 屏幕坐标换算 | AX 返回的是 point 坐标而非像素坐标，`CGEvent` 也用 point，无需额外换算 |
| 3.1.4 | 多显示器环境 | 微信窗口可能在副屏上，坐标系需要正确处理 |

### 3.2 快速滚动消息区获取全部消息

| # | 场景 | 要求 |
|---|------|------|
| 3.2.1 | 消息少（一屏以内） | 不需要滚动，直接读取 |
| 3.2.2 | 消息多（需要滚动才能看到所有未读） | 需要向上滚动加载历史，再回到底部确认无遗漏 |
| 3.2.3 | 滚动速度 | 使用较大 `scroll delta` 加速；每次滚动后等待 UI 更新（约 100-200ms） |
| 3.2.4 | 滚动到顶部后停止 | 检测 `AXScrollArea` 的 `AXValue`（滚动百分比）为 0 时停止 |
| 3.2.5 | 微信的"查看更多消息"按钮 | 有时滚动到顶部会出现"查看更多消息"，需要点击它加载更早的消息 |
| 3.2.6 | 消息加载中的 spinner | 滚动到顶部触发异步加载时会出现 loading 动画，需要等待消失再继续 |

### 3.3 多模态消息类型的识别与提取

| # | 消息类型 | UI / AX 特征 | 提取策略 | `type` 值 |
|---|----------|-------------|----------|-----------|
| 3.3.1 | **纯文本消息** | AXGroup 内含 AXStaticText | 直接读 `AXValue` | `"text"` |
| 3.3.2 | **表情包 / Sticker** | AXImage 元素，无文本 | 记录为 "[表情]" 或提取 AXDescription | `"image"` |
| 3.3.3 | **图片消息** | AXImage 元素，较大尺寸 | 记录为 "[图片]"；可选保存到本地 | `"image"` |
| 3.3.4 | **视频消息** | AXGroup 内含播放按钮图标 | 记录为 "[视频]" | `"unsupported"` |
| 3.3.5 | **语音消息** | AXGroup 内含特定语音条 UI 元素 | 记录为 "[语音]"，提取时长信息（如果有） | `"unsupported"` |
| 3.3.6 | **链接卡片（公众号文章等）** | AXGroup 内含 AXLink 或嵌套 AXStaticText（标题 + 来源） | 提取标题 + 描述 + 来源名称 | `"link_card"` |
| 3.3.7 | **公众号推送消息** | 类似链接卡片，可能有缩略图 AXImage | 提取标题 + 公众号名称 | `"link_card"` |
| 3.3.8 | **视频号分享** | AXGroup 内含视频封面 AXImage + 标题 AXStaticText | 提取标题 + 视频号名称 | `"link_card"` |
| 3.3.9 | **小程序卡片** | AXGroup 内含小程序 icon + 标题 | 提取小程序名称 + 标题 | `"link_card"` |
| 3.3.10 | **文件消息** | AXGroup 内含文件图标 + 文件名 AXStaticText | 提取文件名 + 文件大小 | `"unsupported"` |
| 3.3.11 | **合并转发消息（聊天记录）** | AXGroup 内含多行摘要文本 | 提取摘要文本 | `"text"` |
| 3.3.12 | **名片分享** | AXGroup 内含头像 + 名称 | 提取名称 | `"link_card"` |
| 3.3.13 | **位置分享** | AXGroup 内含地图缩略图 + 地址文本 | 提取地址文本 | `"text"` |
| 3.3.14 | **红包消息** | AXGroup 内含红包 icon + 文案 | 记录为 "[红包] {文案}" | `"text"` |
| 3.3.15 | **转账消息** | AXGroup 内含转账 icon + 金额 | 记录为 "[转账] {金额}" | `"text"` |
| 3.3.16 | **系统消息（时间分隔线、入群通知等）** | 单独 AXStaticText，无发送者头像 | 提取文本，`sender` 设为 `"system"` | `"system"` |
| 3.3.17 | **撤回消息提示** | 系统消息样式，"xxx 撤回了一条消息" | 同系统消息处理 | `"system"` |
| 3.3.18 | **引用/回复消息** | AXGroup 内含引用块（嵌套 AXGroup）+ 回复文本 | 提取引用内容 + 回复内容 | `"text"` |
| 3.3.19 | **拍一拍** | 系统消息样式 | 提取文本 | `"system"` |
| 3.3.20 | **音视频通话记录** | AXGroup 内含通话 icon + 时长/状态 | 记录为 "[通话] {时长/状态}" | `"unsupported"` |

### 3.4 发送者 + 时间戳提取

| # | 场景 | 要求 |
|---|------|------|
| 3.4.1 | 群聊中每条消息有发送者昵称 | 昵称在消息气泡上方的 AXStaticText 中 |
| 3.4.2 | 个人聊天中消息无昵称（只有对方和自己） | 通过消息气泡的左右对齐区分自己和对方 |
| 3.4.3 | 时间戳 | 微信在消息间穿插时间分隔行，提取最近的时间分隔行作为后续消息的时间 |
| 3.4.4 | 消息时间跨天 | 时间分隔行格式变化（"昨天"、"星期X"、"2025年1月1日"等），需要统一格式 |
| 3.4.5 | 群昵称与微信昵称不一致 | 按显示的群昵称为准 |

## 4. 稳定性与容错

| # | 场景 | 预期行为 |
|---|------|----------|
| 4.1 | AX 元素在读取过程中变为 stale（用户切换了聊天） | 捕获 AX 错误码，跳过该元素并 log warning |
| 4.2 | 微信在 pipeline 运行中被关闭 | `find_wechat_window` 在下一次调用时失败，pipeline crash with clear error |
| 4.3 | 微信弹出模态对话框（如更新提示） | 检测到非预期窗口时 log 并 crash |
| 4.4 | 网络断开，消息无法加载 | 正常读取已缓存的消息，无需额外处理 |
| 4.5 | macOS 版本差异导致 AX 树结构变化 | 当关键元素找不到时 crash with descriptive error，方便排查 |
| 4.6 | 单条消息解析失败 | 跳过该消息，标记 `type: "unsupported"`，继续解析其余消息 |

## 5. 输出验证

| # | 检查项 | 要求 |
|---|--------|------|
| 5.1 | JSON 文件格式正确 | 符合 `write_messages_json` 定义的 schema |
| 5.2 | 每条消息都有 `chat_name`, `sender`, `time`, `content`, `type` | 不能有缺失字段 |
| 5.3 | 消息按时间顺序排列 | 从旧到新，与 UI 中从上到下一致 |
| 5.4 | 文件名正确 | `{chat_name}.json`，特殊字符已 sanitize |
| 5.5 | 中文内容无乱码 | `ensure_ascii=False`，UTF-8 编码 |
| 5.6 | 不丢消息 | 读取的消息数 >= 未读 badge 显示的数量 |
| 5.7 | 不重复 | 同一条消息不能出现两次（滚动导致的重复需去重） |
