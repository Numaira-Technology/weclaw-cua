# 未读消息堆成山，不是你的错

**WeClaw 想做的第一件事很简单：用大约十秒钟，帮你读懂几百条未读里真正重要的东西。**

---

工作群、项目群、家长群、兴趣群……红点叠在一起，像一张越收越紧的网。你当然可以把它们全部静音、免打扰，甚至眼不见为净；但你也知道，那种「假装不存在」的轻松，往往伴随着另一种不安：万一错过一条必须回复的 @？一个突然改期的会议？一句客户发来的关键确认？

于是很多人回到老办法：硬读。一条条往上翻，手指划到发酸，时间像水一样流走。问题并不在于你不够自律，而在于**信息的增速早就超过了人肉阅读的合理上限**——继续用「读完」作为标准，本质上是在用昨天的工具，解今天的题。

也有人试过让 AI 介入：装环境、配模型、写脚本、维护一台永远在线的机器。对熟悉工程的人来说，这是乐趣；对只想把消息处理完的人来说，这常常比「自己翻聊天记录」还折磨。我们认同一个朴素的前提：**如果一件产品要求你先成为半个运维，它就很难成为大多数人的默认选择。**

所以我们做了 WeClaw。

它不是又一个陪你闲聊的聊天机器人。更准确地说，它是跑在你**消息场景**里的 AI agent：不抢对话的主角，而是站在洪水一样的未读后面，把它们收成你能一眼扫完的结构——发生了什么、谁在找你、有没有时间节点、哪些事需要你动手。我们希望你在打开总结的那一刻，感觉不是「我又多装了一个工具」，而是「终于有人帮我把噪音和信号分开了」。

上手路径我们也刻意压平：你可以从桌面端开始，把它当作常驻的聊天助手；如果你已经在用 OpenClaw，也可以通过 Skill 的方式接入，让同一套能力出现在你习惯的工作流里。细节命令与参数留给文档；在这里，你只需要记住两件事：**一眼看懂群里发生了什么**，以及**不必自己翻找——该处理的事项会被推到前面**。

这只是一条主线的起点。接下来我们还会往两个方向延伸（落地时会明确标注进度）：一是**消息层面的统计与洞察**——你究竟挂在哪些群里、哪些群在消耗你最多注意力、哪些关系其实可以整理甚至退出；若你有客户或商业场景，高频问题与需求也会逐渐变得可见。二是 **unified command**：多个消息来源，一个入口汇总与总结，让「分散的收件箱」在认知上重新变成「一个待办界面」。

对于更愿意自己掌控数据的用户，本地运行与 Skills 安装依然会是可选路径；我们也正在准备**云服务**形态——让你不必在每一台设备上都装齐各类聊天客户端，也能使用 WeClaw 的能力。（云服务上线前会单独公告。）

如果你愿意现在就看它是否适合你：下载试用，或按文档安装 Skill，跑通一次「从未读到可读总结」的流程。若你认同这种把复杂留给我们、把清晰留给你的方向，欢迎在 GitHub 给项目一颗 Star——这既是对我们的支持，也方便你自己跟踪版本与路线图。想持续了解发版与场景扩展，关注我们的官方渠道即可；我们会在有实质进展时说话，而不是用噪音填满你的时间线。

未读还会再来。但至少，你不必再用整个下午，去证明自己没有错过世界。
